# Vector DB Implementation Spec

The binary vector DB is the agent's unified memory system. It stores memories as 256-bit binary embeddings, retrieves them via hamming distance, and uses Commonware's Journaled MMR as the authenticated persistent storage layer.

## Architecture

The vector DB has two layers:

- **Journaled MMR (host)** — persistent append-only MMR with journal-backed storage. Handles leaf appends, merkleization, proof generation, and disk persistence via `sync()`. This is the source of truth for the MMR commitment.
- **In-memory index (host)** — a `Vec<MemoryEntry>` kept in sync with the MMR. Provides fast iteration for flat-scan queries. Rebuilt from the MMR on startup.

The Jolt guest never touches the Journaled MMR. It receives proofs generated by the MMR and verifies them using lightweight MMR proof verification (`no_std` compatible).

```
Host                                      Jolt Guest (no_std)
┌────────────────────────────────┐        ┌─────────────────────────┐
│ Journaled MMR                  │        │                         │
│  - persistent MMR storage      │        │ MMR proof verification  │
│  - batch → merkleize → apply   │─proof─>│ nonce + sig validation  │
│  - proof generation            │        │ output new state root   │
│                                │        │                         │
│ In-memory index                │        │                         │
│  - Vec<MemoryEntry>            │        └─────────────────────────┘
│  - flat scan (XOR + popcount)  │
└────────────────────────────────┘
```

## Data Structures

### Memory Entry (MMR Leaf)

```rust
struct MemoryEntry {
    id: u64,                // unique, monotonically increasing
    embedding: [u64; 4],    // 256-bit binary vector
    content_hash: [u8; 32], // hash of full content posted as calldata
}
```

72 bytes per entry. Serialized via `commonware-codec` for deterministic hashing. The serialized bytes are what the Journaled MMR stores and hashes as leaves.

### All Mutations Are Appends

The Journaled MMR is append-only. New memories are appended as leaves. Old entries are never modified or removed — the MMR only grows. Relevance is handled at query time via hamming distance search, not by filtering on status flags.

## Embeddings

256 bits stored as `[u64; 4]`. Generated off-chain by either:

1. **Native binary model** (e.g., MixedBread mxbai-embed with binary quantization)
2. **Binarization** of float embeddings (threshold at zero: positive → 1, negative → 0)

Provided as witness data to the guest. The guest does not verify that embeddings represent their content — that's an off-chain trust boundary.

## Hamming Distance

```rust
fn hamming_distance(a: &[u64; 4], b: &[u64; 4]) -> u32 {
    (a[0] ^ b[0]).count_ones()
    + (a[1] ^ b[1]).count_ones()
    + (a[2] ^ b[2]).count_ones()
    + (a[3] ^ b[3]).count_ones()
}
```

4 XOR + 4 popcount. Returns 0 (identical) to 256 (maximally different). For 10,000 entries, a flat scan is 80,000 integer operations — trivial both on the host and inside the Jolt guest.

## Hash Function

Blake3 via `commonware-cryptography::Blake3`. Benchmarked at 4.3x fewer cycles than SHA-256 in the Jolt guest (see `spikes/jolt-merkle/RESULTS.md`), enabling larger batches on lower RAM.

The Journaled MMR uses Commonware's `StandardHasher<Blake3>`, which includes position-aware hashing — leaf digests include their MMR position, preventing second-preimage attacks.

## Batching

The Journaled MMR's batch lifecycle maps directly to our proving strategy:

```
1. Agent handles interactions in real time
2. LLM extracts memories → host appends to in-memory index
3. Appends accumulate via mmr.new_batch() → batch.add()
4. Periodically: batch.merkleize() → finalize() → mmr.apply(changeset)
5. Extract old root, new root, proofs → pass to Jolt guest as witness
6. mmr.sync() persists to disk
7. Guest proves the batch → new vector_index_root posted to L1
```

The agent's live state runs ahead of the proven on-chain state by one batch. Standard rollup behavior.

## Embedding Eras

Embeddings are committed in the MMR leaf, enabling verifiable retrieval. However, embedding models improve over time. A `ReIndex` transition allows the agent to upgrade its embedding model:

1. Re-embed all existing content with the new model
2. Rebuild the MMR from scratch (same content hashes, new embeddings)
3. Commit the new root on-chain as a normal state transition

Content hashes survive across eras — they're model-independent. The `vector_index_root` resets to reflect the new index. This is analogous to an L2 upgrade: state is continuous, only the execution environment changes.

Retrieval proofs are era-scoped: a proof generated before a re-index is valid against the old root, not the new one.

## Guest vs Host

### Host

- Runs Journaled MMR for persistent authenticated storage
- Maintains in-memory index for fast queries
- Generates embeddings via embedding model
- Decides what to store (LLM extracts memories)
- Manages batch lifecycle (new_batch → add → merkleize → apply → sync)
- Extracts proofs from the MMR for the guest

### Guest (Proven in Jolt)

- Verifies old MMR root matches current `vector_index_root`
- Verifies MMR proofs (lightweight `no_std` path verification)
- Verifies the new root is correct given the old root + appended entries
- Outputs new `vector_index_root`

The guest uses `mmr::proof` verification logic only — no Journaled MMR, no `std`, no filesystem.

## Public API

```rust
impl VectorDB {
    // === Construction ===

    /// Create a new vector DB with Journaled MMR persistent storage.
    pub fn new(config: MmrConfig) -> Result<Self>;

    /// Open an existing vector DB from Journaled MMR's persistent storage.
    /// Rebuilds the in-memory index from the MMR on startup.
    pub fn open(config: MmrConfig) -> Result<Self>;

    // === Writes ===

    /// Append a new memory entry. Returns the new MMR root.
    pub fn append(&mut self, entry: MemoryEntry) -> [u8; 32];

    /// Append a batch of entries. Returns the new MMR root.
    pub fn batch_append(&mut self, entries: Vec<MemoryEntry>) -> [u8; 32];

    // === Reads ===

    /// Flat scan: return top-k nearest entries by hamming distance.
    pub fn query(&self, embedding: &[u64; 4], k: usize) -> Vec<QueryResult>;

    /// Get a specific entry by ID.
    pub fn get(&self, id: u64) -> Option<&MemoryEntry>;

    // === Commitment ===

    /// Current MMR root.
    pub fn root(&self) -> [u8; 32];

    // === Witness ===

    /// Produce a witness for the current pending batch (diff only).
    pub fn witness(&self, old_root: [u8; 32]) -> VectorDBWitness;
}

pub struct QueryResult {
    pub entry: MemoryEntry,
    pub distance: u32,
}
```

### Design Decisions

- **`new` vs `open`** — two construction paths: fresh start or resume from local disk (Journaled MMR persists via `sync()`). Reconstruction is done by replaying appends through `new()` + `batch_append()`.
- **Witness contains appends only.** The guest knows the old root. The witness provides new entries and the MMR-generated proofs. Supports batching — a single witness can cover one or many interactions.
- **All write methods return the new root.** The caller always knows the current commitment.

## Recovery Paths

| Scenario | Method | What Happens |
|----------|--------|-------------|
| Normal restart | `open()` | Journaled MMR loads from disk via `init()`, in-memory index rebuilt |
| Reconstruction | Replay from genesis | Replay all transitions via `new()` + `batch_append()`, compare final root to on-chain state |

## Commonware Dependencies

| Crate | Component | Usage |
|-------|-----------|-------|
| `commonware-storage` | `mmr::journaled::Mmr` | Persistent append-only MMR with journal-backed storage |
| `commonware-storage` | `mmr::proof` | Proof verification in the guest (`no_std`) |

## Sizing

| Entries | Entry Storage | Embedding Storage |
|---------|---------------|-------------------|
| 100 | 6.4 KB | 3.1 KB |
| 1,000 | 64 KB | 31 KB |
| 10,000 | 644 KB | 312 KB |
| 100,000 | 6.4 MB | 3.1 MB |

At agent-memory scale (1k-10k entries), everything fits comfortably in memory. The Journaled MMR handles on-disk persistence via `sync()`.
