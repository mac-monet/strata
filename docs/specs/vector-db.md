# Vector DB Implementation Spec

The binary vector DB is the agent's unified memory system. It stores memories as 256-bit binary embeddings, retrieves them via hamming distance, and uses Commonware's QMDB as the authenticated persistent storage layer.

## Architecture

The vector DB has two layers:

- **QMDB (host)** — persistent authenticated storage. Handles append-only data, merkleization, proof generation, and disk persistence. This is the source of truth for the MMR commitment.
- **In-memory index (host)** — a `Vec<MemoryEntry>` kept in sync with QMDB. Provides fast iteration for flat-scan queries and core memory loading. Rebuilt from QMDB on startup.

The Jolt guest never touches QMDB. It receives proofs generated by QMDB and verifies them using lightweight MMR proof verification (`no_std` compatible).

```
Host                                      Jolt Guest (no_std)
┌────────────────────────────────┐        ┌─────────────────────────┐
│ QMDB (keyless)                 │        │                         │
│  - persistent MMR storage      │        │ MMR proof verification  │
│  - batch → merkleize → apply   │─proof─>│ nonce + sig validation  │
│  - proof generation            │        │ output new state root   │
│                                │        │                         │
│ In-memory index                │        │                         │
│  - Vec<MemoryEntry>            │        └─────────────────────────┘
│  - flat scan (XOR + popcount)  │
│  - core memory filtering       │
└────────────────────────────────┘
```

## Data Structures

### Memory Entry

```rust
struct MemoryEntry {
    id: u64,                // unique, monotonically increasing
    embedding: [u64; 4],    // 256-bit binary vector
    content_hash: [u8; 32], // hash of full content posted as calldata
    core: bool,             // true = always loaded into agent context
    active: bool,           // false = soft deleted (superseded by a newer entry)
}
```

Serialized via `commonware-codec` for deterministic hashing. The serialized bytes are what QMDB stores and hashes as MMR leaves.

### All Mutations Are Appends

QMDB's keyless variant is append-only, which matches our mutation model:

- **New memory** → append entry
- **Modified memory** → append new version, mark old entry `active: false`
- **Deleted memory** → append tombstone with `active: false`

The in-memory index reflects these changes. Old entries are never removed from QMDB — the MMR is append-only. The `active` flag controls what appears in queries.

## Embeddings

256 bits stored as `[u64; 4]`. Generated off-chain by either:

1. **Native binary model** (e.g., MixedBread mxbai-embed with binary quantization)
2. **Binarization** of float embeddings (threshold at zero: positive → 1, negative → 0)

Provided as witness data to the guest. The guest does not verify that embeddings represent their content — that's an off-chain trust boundary.

## Hamming Distance

```rust
fn hamming_distance(a: &[u64; 4], b: &[u64; 4]) -> u32 {
    (a[0] ^ b[0]).count_ones()
    + (a[1] ^ b[1]).count_ones()
    + (a[2] ^ b[2]).count_ones()
    + (a[3] ^ b[3]).count_ones()
}
```

4 XOR + 4 popcount. Returns 0 (identical) to 256 (maximally different). For 10,000 entries, a flat scan is 80,000 integer operations — trivial both on the host and inside the Jolt guest.

## Hash Function

SHA-256 via `commonware-cryptography::Sha256`. Can swap to Blake3 if benchmarking shows better proving time. Both are available in Commonware.

QMDB uses Commonware's MMR hasher (`mmr::hasher::Standard<Sha256>`), which includes position-aware hashing — leaf digests include their MMR position, preventing second-preimage attacks.

## Batching

QMDB's batch lifecycle maps directly to our proving strategy:

```
1. Agent handles interactions in real time
2. LLM extracts memories → host appends to in-memory index
3. Appends accumulate in a QMDB UnmerkleizedBatch
4. Periodically: batch.merkleize() → MerkleizedBatch (has new root, can generate proofs)
5. Extract old root, new root, proofs → pass to Jolt guest as witness
6. batch.finalize() → Changeset → db.apply() (persist to disk)
7. Guest proves the batch → new vector_index_root posted to L1
```

The agent's live state runs ahead of the proven on-chain state by one batch. Standard rollup behavior.

## Guest vs Host

### Host

- Runs QMDB for persistent authenticated storage
- Maintains in-memory index for fast queries
- Generates embeddings via embedding model
- Decides what to store (LLM extracts memories)
- Decides core vs non-core tagging
- Manages batch lifecycle (accumulate → merkleize → finalize → apply)
- Extracts proofs from merkleized batches for the guest

### Guest (Proven in Jolt)

- Verifies old MMR root matches current `vector_index_root`
- Verifies MMR proofs from QMDB (lightweight `no_std` path verification)
- Verifies the new root is correct given the old root + appended entries
- Outputs new `vector_index_root`

The guest uses `mmr::proof` verification logic only — no QMDB, no `std`, no filesystem.

## Public API

```rust
impl VectorDB {
    // === Construction ===

    /// Create a new vector DB with QMDB persistent storage.
    pub fn new(config: QmdbConfig) -> Result<Self>;

    /// Open an existing vector DB from QMDB's persistent storage.
    /// Rebuilds the in-memory index from QMDB on startup.
    pub fn open(config: QmdbConfig) -> Result<Self>;

    // === Writes ===

    /// Append a new memory entry. Returns the new MMR root.
    pub fn append(&mut self, entry: MemoryEntry) -> [u8; 32];

    /// Append a batch of entries. Returns the new MMR root.
    pub fn batch_append(&mut self, entries: Vec<MemoryEntry>) -> [u8; 32];

    // === Reads ===

    /// Flat scan: return top-k nearest active entries by hamming distance.
    pub fn query(&self, embedding: &[u64; 4], k: usize) -> Vec<QueryResult>;

    /// Return all active core memory entries.
    pub fn core_memories(&self) -> Vec<&MemoryEntry>;

    /// Get a specific entry by ID (if active).
    pub fn get(&self, id: u64) -> Option<&MemoryEntry>;

    // === Commitment ===

    /// Current MMR root.
    pub fn root(&self) -> [u8; 32];

    // === Witness ===

    /// Produce a witness for the current pending batch (diff only).
    pub fn witness(&self, old_root: [u8; 32]) -> VectorDBWitness;
}

pub struct QueryResult {
    pub entry: MemoryEntry,
    pub distance: u32,
}
```

### Design Decisions

- **`new` vs `open`** — two construction paths: fresh start or resume from local disk (QMDB persists). Reconstruction is done by replaying appends through `new()` + `batch_append()`.
- **Witness contains diffs only.** The guest knows the old root. The witness provides appends, deactivations, and the QMDB-generated proofs. Supports batching — a single witness can cover one or many interactions.
- **All write methods return the new root.** The caller always knows the current commitment.

## Recovery Paths

| Scenario | Method | What Happens |
|----------|--------|-------------|
| Normal restart | `open()` | QMDB loads from disk, in-memory index rebuilt from QMDB |
| Reconstruction | Replay from genesis | Replay all transitions via `new()` + `batch_append()`, compare final root to on-chain state |

## Commonware Dependencies

| Crate | Component | Usage |
|-------|-----------|-------|
| `commonware-storage` | `qmdb::keyless` | Persistent authenticated append-only storage |
| `commonware-storage` | `mmr::proof` | Proof verification in the guest (`no_std`) |
| `commonware-storage` | `mmr::hasher::Standard` | Position-aware MMR hashing |
| `commonware-cryptography` | `Sha256` (or `Blake3`) | Hash function for leaves and nodes |
| `commonware-codec` | `Write`, `Read`, `FixedSize` | Deterministic serialization of `MemoryEntry` |

## Sizing

| Entries | Entry Storage | Embedding Storage |
|---------|---------------|-------------------|
| 100 | 6.4 KB | 3.1 KB |
| 1,000 | 64 KB | 31 KB |
| 10,000 | 644 KB | 312 KB |
| 100,000 | 6.4 MB | 3.1 MB |

At agent-memory scale (1k-10k entries), everything fits comfortably in memory. QMDB handles the on-disk persistence transparently.
